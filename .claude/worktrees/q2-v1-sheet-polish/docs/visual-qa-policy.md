# StudyHub Visual QA Policy

## Rule

**No user-facing feature, bug fix, or UI refactor is complete until it has visual proof.**

That means every meaningful change must include:
- screenshots
- state coverage
- theme coverage
- viewport coverage when relevant

---

## 1) What requires visual proof

### Always required
- new pages
- redesigned pages
- new components
- UI bug fixes
- dark mode changes
- responsive/mobile fixes
- empty states
- modals/drawers/dialogs
- admin tools
- upload/preview/review flows

### Usually required
- visible content/status changes
- new banners/badges
- onboarding flows
- filters/search UI
- settings/profile changes

### Not required
- backend-only changes with no UI effect
- docs-only changes
- pure refactors with zero UI change
  *(though one smoke screenshot is still nice if behavior changed indirectly)*

---

## 2) Required screenshot set

### Minimum standard
For every change:
- **Before**
- **After**

### Standard UI coverage
For each affected screen:
- **Light mode**
- **Dark mode**
- **Desktop**
- **Mobile** if the page is responsive or user-facing

### State coverage
Capture the relevant states:
- empty
- populated
- loading
- success
- warning
- error
- modal open / drawer open if applicable

---

## 3) Required naming convention

### Format
```txt
{surface}-{state}-{theme}-{viewport}.png
```

### Examples
```txt
upload-empty-light-desktop.png
upload-flagged-dark-desktop.png
sheets-pending-light-mobile.png
admin-review-pending-dark-desktop.png
profile-error-light-desktop.png
feed-post-with-image-dark-mobile.png
```

### For before/after comparisons
```txt
before-upload-empty-light-desktop.png
after-upload-empty-light-desktop.png
```

or:

```txt
upload-empty-light-desktop-before.png
upload-empty-light-desktop-after.png
```

Pick one style and keep it consistent.

---

## 4) Folder structure

### Recommended structure
```txt
docs/release-visuals/
  cycle-38/
    upload/
    sheets/
    admin/
    profile/
    feed/
```

### Example
```txt
docs/release-visuals/
  cycle-39/
    upload/
      before-upload-empty-light-desktop.png
      after-upload-empty-light-desktop.png
      after-upload-flagged-dark-desktop.png
    admin/
      before-admin-review-light-desktop.png
      after-admin-review-light-desktop.png
    sheets/
      after-sheets-pending-light-mobile.png
```

### Optional test-results mirror
If generated by Playwright:
```txt
frontend/studyhub-app/test-results/visuals/cycle-39/
```

You can store raw captures there and copy curated ones into `docs/release-visuals/`.

---

## 5) Per-feature visual checklist

Every feature/change should answer:

### A. What did it look like before?
- screenshot captured

### B. What does it look like after?
- screenshot captured

### C. Does it work in light mode?
- screenshot captured

### D. Does it work in dark mode?
- screenshot captured

### E. Does it work on mobile?
- screenshot captured if applicable

### F. Did we capture the important state?
Examples:
- warning banner
- empty state
- pending state
- modal
- error message
- success flow

---

## 6) Workflow-specific screenshot requirements

### Upload / HTML workflow
Must capture:
- upload page blank state
- imported/edited state
- flagged scan state
- pending review state
- preview page
- admin review panel

### Sheets page
Must capture:
- default state
- mine state
- pending filter
- empty state
- mobile state

### Feed
Must capture:
- normal feed card
- post with image
- post attachment preview
- dark mode feed card
- mobile feed layout

### Profile
Must capture:
- normal profile
- avatar loaded
- avatar fallback
- error state
- mobile profile

### Admin
Must capture:
- queue list
- review panel
- warning/high-risk state
- action area
- dark mode if supported

---

## 7) Visual gallery requirement

Each cycle should produce **one visual gallery**.

### Gallery should include
- before/after pairs
- key feature shots
- major warning/error states
- mobile + dark mode highlights

### Goal
Anyone reviewing the cycle should be able to answer:
- what changed?
- does it look better?
- does it still work?
- does it behave correctly in major states?

---

## 8) Release requirement

A cycle is not fully closed until it has:

- code complete
- tests complete
- lint/build passing
- docs updated
- **visual gallery attached**

---

## 9) Quality bar

### Good visual proof
- clean crop
- consistent viewport
- obvious state being shown
- readable text
- not redundant
- easy to compare

### Bad visual proof
- random screenshots
- no before/after
- only one theme
- only one viewport
- missing warning/error state
- screenshots with no naming structure

---

## 10) Operating rule

> **No visible feature, redesign, or UI bug fix is considered done until we have before/after screenshots and theme/state coverage for the affected surface.**

---

## 11) Cycle template addition

Add this section to every cycle report:

### Visual Proof
- Before screenshots captured
- After screenshots captured
- Light mode checked
- Dark mode checked
- Mobile checked where applicable
- Gallery generated
- Stored under: `docs/release-visuals/cycle-XX/`

---

## 12) Per-cycle track

For every cycle with UI changes, add a dedicated item:

### Track V -- Visual QA + Gallery
Deliverables:
- screenshot set for every track
- curated gallery
- before/after proof for all visible changes
